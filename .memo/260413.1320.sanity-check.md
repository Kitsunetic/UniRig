# ✅ UniRig Full Validation + Evaluation 체크리스트 프롬프트

## 🎯 목표

기존 UniRig repository 코드를 그대로 사용하여, 아래 항목을 end-to-end로 검증해줘.

1. dataloader 동작 확인
2. model 생성
3. pretrained weight 다운로드 및 `load_state_dict(strict=True)` 검증
4. RTX 3090 1개 환경에서 training sanity check
5. inference 수행
6. qualitative 결과 확인
7. metric 계산을 통한 정량 평가
8. benchmark / README / 논문 / 공식 결과와 비교

---

# 🔒 작업 원칙

* **새로운 model / dataloader / trainer / evaluator를 구현하지 말 것**
* repository에 이미 있는 **기존 구현, config, script, builder, evaluator**를 최대한 그대로 사용할 것
* 기존 코드를 수정하지 말고, **검증용 코드만 `src/test/` 아래에 추가**할 것
* 기존 evaluator가 있으면 그것을 우선 사용하고, 정말 없을 때만 최소한의 보조 계산 코드를 작성할 것
* 목적은 성능 개선이 아니라 **재현 가능성, 정합성, 실행 가능성, 평가 가능성 검증**이다

# 1. 코드 구조 파악

먼저 아래 위치를 찾아 정리해줘.

* dataset / dataloader 정의 위치
* model 정의 위치
* model builder 또는 config 기반 생성 경로
* pretrained loading 관련 코드
* training entrypoint
* inference entrypoint
* evaluation / metric 계산 코드
* benchmark 또는 공식 성능 수치가 적혀 있는 문서 위치

## 확인할 것

* 어떤 config가 pretrained와 연결되는지
* 어떤 split이 evaluation용인지
* 어떤 metric이 공식 metric인지

## 출력

* 관련 파일 경로 목록
* 실제 사용할 entrypoint / config 파일 목록
* evaluation에 사용되는 split 이름

# 2. 데이터셋 및 dataloader 검증

이미 일부 검증되었다고 가정하되, 아래를 다시 확인해줘.

## 확인 항목

* dataset이 정상적으로 로드되는지
* split(train / val / test)별 샘플 수
* sample / batch의 key 이름
* 각 key의 shape / dtype

## 출력

* 전체 데이터 개수
* split별 개수
* 샘플 구조 예시
* datalist와 mapping.json이 어떻게 사용되는지 간단 설명

---

# 3. pretrained weight 확보 및 구조 확인

pretrained weight는 직접 다운로드해줘.

## 우선순위

1. repository README
2. release / script / config
3. 공식 문서나 명시된 링크

## 확인 항목

* 다운로드 URL
* 저장 경로
* checkpoint 내부 구조

  * 예: `state_dict`, `model`, `module`, `ema`, `net` 등
* 어떤 config와 짝이 맞는 checkpoint인지

## 출력

* URL
* 저장 파일명
* checkpoint top-level key 구조
* 실제 state_dict 추출 방법

---

# 4. model 생성 및 strict 로딩 검증

pretrained와 일치하는 config로 model을 생성해줘.

## 수행

* model instantiate
* total params / trainable params 출력
* `load_state_dict(strict=True)` 시도

## 성공 조건

* strict=True로 에러 없이 로딩 성공

## 실패 시 반드시 분석

* missing keys
* unexpected keys
* shape mismatch
* prefix mismatch (`module.` 등)
* config 불일치 가능성

## 출력

* strict load 성공 여부
* 주요 key 5~10개 비교
* mismatch 원인 분석

---

# 5. training sanity check

환경은 **RTX 3090 GPU 1개**다.
single GPU 기준으로 기존 training pipeline이 실제 동작하는지 검증해줘.

## 수행 항목

* model을 GPU로 이동
* batch 로드
* forward
* loss 계산
* backward
* optimizer step
* 가능하면 scheduler step도 확인

## 실행 범위

* 최소 3 step, 권장 5~10 step
* OOM 방지를 위해 batch size는 보수적으로 시작
* 기존 AMP가 있으면 사용 가능
* DDP / multi-GPU는 사용하지 말 것

## 확인 항목

* loss가 정상 숫자인지
* NaN / Inf 여부
* gradient update 정상 여부
* optimizer / scheduler 에러 여부
* pretrained 로드 후 학습 가능 여부

## 출력

* device
* batch size
* step 수
* step별 loss
* OOM / NaN 발생 여부

---

# 6. inference 검증

pretrained weight를 로드한 model로 inference를 수행해줘.

## 수행

* `model.eval()` 설정
* 샘플 1개 또는 batch 1개 입력
* forward 결과 확인

## 확인 항목

* output key 이름
* output tensor shape
* output dtype
* inference 중 에러 여부

## 추가 권장

* 동일 입력 2회 inference하여 deterministic 여부 또는 큰 차이 없는지 확인

## 출력

* input 구조
* output 구조
* shape / dtype
* 동일 입력 반복 결과 비교 요약

---

# 7. qualitative 결과 확인

모델 출력이 실제로 말이 되는지 sanity level에서 확인해줘.

## 출력 형태에 맞춰 확인

다음 중 해당되는 항목을 적용:

* joint positions
* rig / skeleton
* mesh vertices
* skinning weights
* point cloud
* 이미지
* feature vector

## 확인 항목

* 값이 전부 0인지
* NaN / Inf 포함 여부
* 값 범위가 비정상적으로 큰지
* 구조적으로 plausible한지

## 가능한 경우

* sample prediction 일부 저장
* 좌표 일부 출력
* min / max / mean 요약
* mesh / joint 개수 출력
* 이미지가 있으면 시각화 결과 저장

## 출력

예시:

```text
prediction keys: ...
joints shape: (N, 3)
vertices shape: (M, 3)
weights shape: (M, K)
min/max/mean: ...
first sample values: ...
```

## 결론

* qualitative 관점에서 정상 / 의심 / 실패 중 하나로 판정

# 8. evaluation metric 계산

이제 정량 평가를 수행해줘.

## 원칙

* repository에 이미 구현된 evaluator / metric 계산 코드가 있으면 그것을 최우선 사용
* 공식 metric 정의를 먼저 확인
* 공식 evaluator가 없을 때만 최소한의 계산 코드 추가

## 해야 할 일

* evaluation split 결정
* inference 결과를 evaluator 입력 형식에 맞게 연결
* metric 계산 실행

## 확인 항목

* 어떤 metric이 사용되는지
* metric 정의가 어디에 있는지
* split 전체를 평가했는지, subset만 평가했는지
* subset이면 왜 subset인지

## 출력

* 사용한 metric 이름
* metric 정의 위치
* evaluation split
* 평가 샘플 수
* 산출된 metric 값

---

# 9. benchmark / 공식 결과와 비교

평가 결과를 README, 논문, release note, 공식 문서에 있는 기준 성능과 비교해줘.

## 수행

* 공식 benchmark 수치 탐색
* 현재 실행 결과와 비교
* 차이를 해석

## 반드시 확인할 것

* 동일한 dataset split인지
* 동일한 config / checkpoint인지
* 동일한 evaluation protocol인지
* 동일한 metric 정의인지

## 출력

* 공식 성능 수치 출처 위치
* 재현 결과
* 차이값
* 차이에 대한 가능한 원인 분석

예:

* evaluation subset 차이
* preprocessing 차이
* config 불일치
* checkpoint 버전 차이
* seed / AMP / batch size 영향
* 일부 postprocess 미적용
